## 工作流概述

当用户说"帮我做一个 XX 系统/功能/项目"时，按以下 5 个阶段推进：

```
需求澄清 → 写 OpenSpec → 用户确认 → 分 Phase 编码 → 跨 session 续接
```

---

## 阶段 1：需求澄清

**在调用任何工具之前**，通过多轮对话逐步厘清需求。**不要急于进入 Spec 编写**——复杂系统的需求通常需要多轮才能足够清晰。

### 文档附件优先策略

**如果用户同时提交了需求文档（Word/PDF/PPT/图片），优先解析文档，再提问补充信息：**

| 文件类型 | 解析方法 |
|---------|---------|
| `.docx` / `.doc` | `pandoc document.docx -o requirement.md` 提取全文（需安装 pandoc） |
| `.pdf` | `pip install pdfplumber` → Python 提取文本+表格 |
| `.pptx` | `pip install "markitdown[pptx]"` → `markitdown presentation.pptx`（Microsoft 开源，本地运行，无需 LLM） |
| 设计图 `.png/.jpg` | 直接用 `read_image` 工具分析界面设计 |
| 多个文件混合 | 逐一解析，综合信息后再提问 |

文档解析完成后：
1. 输出"我从文档中理解到的需求摘要"（功能列表、用户角色、约束等）
2. 指出文档中**未说清楚**的信息，只针对这些缺口提问
3. 不要重复问文档中已经明确的内容

### 澄清节奏（纯对话，无文档时）

- **每次只问 1-2 个最关键的问题**，不要一次把所有问题堆上去
- 用户回答后，立即追问下一个最关键的未知点，或转向下一个维度
- **持续多轮对话，直到用户主动说"可以开始了"/"就这样吧"/"开始写吧"** 才进入阶段 2
- 如果用户说"你觉得呢"/"帮我决定"，给出建议并说明理由，然后继续问下一个问题

### 必须澄清的维度（按优先级）

**第一批：项目基础（首轮必问）**
1. **工程目录**：代码和 Spec 文件放在哪个本地目录？（必须明确，所有文件都会写到这里）
   - 示例：`D:\projects\my-app` 或 `~/projects/my-app`
   - 如果用户不确定，建议一个路径并等用户确认
2. **目标用户**：主要给谁用？内部工具 / 个人使用 / 面向用户产品？预期同时在线人数？
3. **核心功能**：用一句话描述最核心的价值是什么？列出不可缺少的 3 个功能。
4. **MVP 边界**：哪些功能可以 v1.0 先不做？（帮助控制范围）

**第二批：技术选型**
5. **后端语言**：Python / Node.js / Go / Rust / Java / 其他？或纯前端无后端？
6. **前端**：React / Vue / 原生 HTML / 命令行 / 无前端？
7. **数据库**：SQLite / PostgreSQL / MySQL / MongoDB / Redis / 无？
8. **运行环境**：本地运行 / 云服务器 / Docker 容器 / Tauri 桌面应用？操作系统？

**第三批：细节与约束**
9. **认证/权限**：需要用户登录吗？多角色权限吗？
10. **集成依赖**：需要对接哪些外部服务或 API？（邮件、支付、第三方登录等）
11. **性能要求**：有没有明显的性能瓶颈点？（大文件处理、高并发、实时推送？）
12. **质量要求**：快速原型（跑起来即可）还是生产就绪（需要完整错误处理、日志、测试）？

**第四批（针对复杂系统）**
13. **数据规模**：预计数据量级？（百条 / 万条 / 百万条？）
14. **已有代码**：有没有已存在的代码需要集成或参考？路径在哪？
15. **参考系统**：有没有想参考的已有产品？（可以指导 UI 和交互设计）

### 规则

- 已经明确的信息不重复问（包括从文档中已提取的信息）
- 用户的回答引出新问题时，继续追问，不要因为"问得够多了"而停止
- **工程目录**是唯一必须在第一轮就确认的信息（文档里通常不会写这个）
- 进入阶段 2 的唯一触发条件：用户明确表示"可以开始了"或对提出的方案表示认可
- 有文档时：从"我理解的需求是..."开始，然后只问缺口，比纯对话少问很多

## 阶段 2：生成 OpenSpec

用户确认"可以开始了"后，用 `write_file` 在**用户指定的工程目录**中创建完整结构：

```
<用户指定的工程目录>/              ← 阶段1中用户确认的路径
  openspec/
    changes/<feature-name>/
      proposal.md     ← 需求描述 + 用户故事 + 验收标准
      design.md       ← 技术选型 + 架构 + 关键决策
      tasks.md        ← Phase 分解（用 checkbox）
      specs/
        api/spec.md   ← 接口清单（如有后端 API）
        data/spec.md  ← 数据模型
        ui/spec.md    ← 页面/组件清单（如有前端）
  src/                ← 后续阶段4中所有代码写到这里
  README.md           ← 项目简介（简版）
```

所有后续的代码文件（阶段 4）也写入这个工程目录，保持 spec 与代码在同一根目录下。

### proposal.md 模板

```markdown
# Proposal: <功能名>

## 背景与目标
[用户的原始需求 + 为什么要做]

## 用户故事
- US-1 ...
- US-2 ...

## 验收标准
- [ ] ...
- [ ] ...

## 不在范围内
- ...
```

### tasks.md 模板

```markdown
# Tasks: <功能名>

> 最后更新：<日期> | 当前进度：尚未开始

## Phase 1：<后端/基础框架>
- [ ] 1.1 ...
- [ ] 1.2 ...

## Phase 2：<核心逻辑>
- [ ] 2.1 ...

## Phase 3：<前端/集成>
- [ ] 3.1 ...

## Phase 4：<测试 + 文档>
- [ ] 4.1 运行测试套件
- [ ] 4.2 更新 README
```

**Phase 划分原则**：
- 每个 Phase 约 20-40 个工具调用（约 1 次对话）
- Phase 1 优先建立可运行的骨架（哪怕功能不完整）
- 最后一个 Phase 必须包含测试和文档

## 阶段 3：用户确认

生成 Spec 后，输出以下摘要等待确认：

```
📋 Spec 已生成：openspec/changes/<feature>/

**项目概述**：[一句话描述]

**技术栈**：[后端] + [前端] + [数据库]

**开发计划**（共 N 个 Phase）：
- Phase 1：[描述] → 预计 XX 个文件
- Phase 2：[描述]
- ...

**第一个 Phase 将完成**：[具体内容]

回复"开始"或"确认"启动 Phase 1，或提出修改意见。
```

---

## 阶段 4：分 Phase 编码

### 每个 Phase 的执行规则

1. **开始前**：读取 `tasks.md`，确认当前 Phase 的所有 tasks
2. **执行中**：
   - 写代码用 `write_file`（新文件）或 `edit_file`（修改已有文件）
   - 每写完一个模块，立即用 `bash` 运行（编译检查 / 单元测试）
   - 发现依赖缺失：用 `bash` 安装，不要停下来问用户
3. **结束时**（必须做）：
   - 用 `edit_file` 更新 `tasks.md`：已完成的 task 改为 `[x]`，更新进度行
   - 调用 `memory_save` 保存 checkpoint（见格式）
   - 输出 Phase 完成摘要

### Phase 完成摘要格式

```
✅ Phase X 完成

**做了什么**：[一句话]

**已完成**：
- [关键文件或功能]

**遇到的问题**（如有）：[简述 + 如何解决的]

**下一 Phase（Phase X+1）**：[描述内容]

回复"继续"开始 Phase X+1。
```

### 工具调用上限保护

若剩余工具调用次数 < 20（通过已使用次数估算），**主动停止**当前 Phase：

```
⚠️ 本次对话工具调用次数接近上限，主动暂停。

**已完成**：[已做的 tasks，打 ✓]
**未完成**：[剩余 tasks]

已保存进度到 tasks.md。下次对话说"继续 Phase X"即可续接。
```

### memory_save checkpoint 格式

```
[project-dev] feature=<name>
project-dir=<用户工程目录绝对路径>
phase=<当前Phase编号>/<总Phase数>
status=in-progress|blocked|done
spec=<工程目录>/openspec/changes/<name>/tasks.md
last-action: <一句话描述最后完成的操作>
blocking: <阻塞原因（如有）>
```

---

## 阶段 5：跨 session 续接

当用户在新 session 中说"继续上次的项目"/"继续开发"/"继续 Phase X" 时：

1. 用 `memory_search` 搜索 `[project-dev]` 找到最近的 checkpoint，提取 `project-dir` 和 `spec` 路径
2. 用 `read_file` 读取 checkpoint 中记录的 `tasks.md` 绝对路径，找到第一个未完成的 Phase
3. 输出状态摘要：

```
📌 续接上次项目：<feature-name>

**当前进度**：Phase X / 总 N（<描述>）
**待完成**：
- [ ] X.Y ...
- [ ] X.Z ...

继续执行 Phase X？
```

4. 用户确认后继续执行，遵循阶段 4 的规则

---

## 注意事项

- **Windows PowerShell 环境**：`bash` 工具实际执行 PowerShell，多行脚本必须先 `write_file` 保存再执行
- **大型前端项目**：如果有 `npm install`，放在 Phase 1 最开始做，后续 Phase 不重复安装
- **数据库迁移**：每次 schema 变更，在 `specs/data/spec.md` 更新数据模型定义
- **测试优先**：Phase 4（最终测试）不是可选的，每个核心模块必须有至少一个可运行的验证步骤
