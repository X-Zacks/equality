# 执行摘要

本文将 Anthropic 的 Skills 体系视为 Agent
开发中的"基础服务层"，并把它与传统软件工程中的函数库、中间件、框架能力以及可复用模块进行类比分析。与"单个大模型 +
单个 prompt"的实现方式不同，Skills
将复杂任务拆分为可动态装载、可复用、可治理、可组合的能力单元。官方仓库将
Skills 定义为由 instructions、scripts 与 resources 组成的文件夹，Claude
会在需要时动态加载，用于提升特定任务的性能与稳定性；官方同时说明，这些技能覆盖创意设计、开发与技术任务、企业沟通以及文档处理等场景，并已经能在
Claude Code、Claude.ai 以及 API 中使用。

从官方仓库与社区 collection 的对比来看，Anthropic
官方仓库更像"参考实现 + 标准样板 +
生产级复杂技能的示例集合"，其优势在于权威、设计清晰、能力边界明确；社区
collection 则扮演"生态目录 + 发现层 + 长尾补充层"的角色，展示了 173
个技能、13
个类别，既包含官方能力，也包含大量社区扩展，对我们理解生态广度和未来企业落地方向非常有价值，但其中技能的质量、许可、维护活跃度和安全性并不一致。

如果把 Skills 看成 Agent 的基础服务层，那么企业构建员工日常使用的
Agent，不应只停留在"文档技能有哪些"这种表层清单上，而应进一步设计一套分层架构：底层是基础工具技能与执行环境；中层是领域技能、企业规范技能和数据技能；上层是由多个
Skills
编排而成的复合流程技能；最上层是面向员工场景的产品化工作流与治理层。这样的结构既能复用已有技能，也能让模型选择、上下文管理、身份权限、知识记忆、审计和用户体验形成统一架构。

本文的核心判断包括五点。第一，Anthropic Skills
的真正价值不在于"单个技能多强"，而在于它把模型能力从一次性回答推进到可组合、可动态加载的工作单元。第二，官方仓库中的
document skills（docx、pdf、pptx、xlsx）与 enterprise / communication /
development skills 已经显示出较清晰的基础层与应用层关系。第三，社区
collection 进一步证明了 Skills
生态正在向生产力工具、项目协作、营销、研究、自动化和安全测试等方向扩展。第四，面向企业员工的
Skills
体系不宜直接照搬社区清单，而应建立"基础能力层---通用办公层---企业知识与流程层---高阶编排层"的分层架构。第五，Skills
的治理比技能本身更重要：来源可信性、版本管理、上下文隔离、权限边界、模型路由、可观测性和审计能力，决定了它是否能在企业环境中成为真正可用的
Agent 基础设施。

# 一、研究对象与方法：为什么要把 Skills 当作 Agent 的基础服务层

如果把传统软件工程与 Agent 工程对比，Prompt 更像一次函数调用，而 Skills
更像可被重复装载的模块化库。Prompt 负责当前任务的指令表达，Skills
负责把某类任务的做法、约束、示例、脚本、模板、资源和工具调用方式固化下来。Anthropic
官方在 README 中明确指出：Skills 是"folders of instructions, scripts,
and resources that Claude loads dynamically to improve performance on
specialized tasks"。这一点很关键，因为它说明 Skills
并不是简单的文本模板，而是一套能够被运行时按需装载的能力包。

从工程视角看，Skills
解决了三个问题。第一，重复任务的稳定复现问题。把经验固化为 Skill
之后，模型不必每次从零推理。第二，复杂任务的组织问题。通过
instructions、脚本与资源组合，模型获得了更像"领域能力"的工作单元，而不是零散工具。第三，团队协作和治理问题。Skills
作为文件夹和标准格式，天然适合版本管理、审阅、发布、测试和权限控制。

本文采用"两层来源、三段分析"的方法。两层来源包括：其一，Anthropic
官方仓库 anthropics/skills，用于分析官方对 Skills
的定义、参考实现和官方推荐能力边界；其二，社区仓库
claude-skills-collection，用于分析生态广度、常见分类和长尾技能方向。三段分析包括：第一段是"看清单"，梳理
Skills
的类别和代表性能力；第二段是"看关系"，分析基础调用、组合关系和高阶使用方式；第三段是"看架构"，把这些观察抽象成面向企业员工日常使用的
Skills 分层设计。

# 二、Anthropic 官方 Skills 的结构与价值

Anthropic 官方仓库给出的信号非常明确。首先，仓库不是单纯的 demo
集合，而是官方对 Skills 体系的一种公开展示。README 明确写到：该仓库包含
Anthropic 自己对 Claude Skills 的实现，同时列出了 What are skills、Using
skills in Claude、How to create custom skills 和 Skills API Quickstart
等入口。这说明 Skills 不是旁支能力，而是已经进入产品、开发框架与 API
的统一能力机制。

其次，官方仓库本身呈现出比较清晰的技能谱系。仓库 skills 目录下可看到
algorithmic-art、brand-guidelines、canvas-design、claude-api、doc-coauthoring、docx、frontend-design、internal-comms、mcp-builder、pdf、pptx、skill-creator、slack-gif-creator、theme-factory、web-artifacts-builder、webapp-testing、xlsx
等文件夹。从命名和组织方式看，官方已经把 Skills
划分为几类：一类是文档处理类，典型如 docx / pdf / pptx /
xlsx；一类是设计与创意类，典型如
algorithmic-art、canvas-design、theme-factory；一类是开发与技术类，典型如
webapp-testing、mcp-builder、claude-api；还有一类是企业沟通与组织规范类，例如
brand-guidelines、internal-comms、doc-coauthoring。

第三，官方仓库对"生产级技能"和"参考级技能"做了边界划分。README
明确指出，很多技能是 Apache 2.0 开源，但 docx、pdf、pptx、xlsx 这些支撑
Claude 文档能力的技能是 source-available、not open
source。这一做法透露出一个重要事实：当 Skill
足够复杂、足够接近生产应用时，它已经不只是一个轻量指令模板，而更像产品能力背后的核心工程模块。也正因此，这四个文档技能特别值得作为企业员工
Agent 的基础服务层样板进行研究。

最后，官方仓库还表明 Skills 具备跨产品一致性。README 写明这些 Skills
可在 Claude Code、Claude.ai 和 Claude API 中使用。这意味着 Skills
并非某个单点产品的内置插件，而更接近 Anthropic 试图统一 agentic
experience 的能力接口。这也是我们将其视为基础服务层的根本原因。

# 三、常用 Skills 分类：从官方仓库到社区生态

从"企业员工日常使用"的角度，Anthropic Skills 可以分为六大类。

第一类是文档与内容生产类。这是最成熟、最容易形成生产力闭环的部分。官方的
docx、pdf、pptx、xlsx 基本对应 Word、PDF、PPT、Excel
四大办公载体。它们适合承载报告撰写、文档整理、表格计算、演示材料生成、审阅和批注等高频办公任务。社区
collection 也把 Document Skills 单列出来，并在文档技能之外扩展了
revealjs-skill 等演示文稿生成能力，说明这一层非常适合作为企业员工 Agent
的默认能力层。

第二类是企业规范与沟通类。官方仓库中的
brand-guidelines、internal-comms、doc-coauthoring 说明 Skills
不只用于"生成文件"，也可用于"规范输出方式"。这类技能的本质是把企业模板、品牌语调、沟通风格、评审流程和协同规则编码为可复用的工作单元。对企业来说，这一层的价值极高，因为它能让
Agent 输出更贴近组织表达，而不是泛化的模型文字。

第三类是开发与技术支持类。官方仓库中的
mcp-builder、webapp-testing、claude-api、skill-creator
展示出一条明显路径：Skills 本身也可以用于帮助开发新的 Skills、开发
MCP、测试 Web 应用和调用 API。也就是说，Skills
体系并不是只服务终端用户，它也服务 Agent
平台工程师和扩展开发者。这一点和传统软件里的"自举工具链"很类似。

第四类是创意与设计类。官方提供
algorithmic-art、canvas-design、slack-gif-creator、theme-factory、frontend-design、web-artifacts-builder
等能力，说明 Skills
也被用来结构化地产生视觉资产、前端页面与交互物料。这类能力对员工工作虽不是最高频，但在营销、设计、培训、内部宣传、产品原型等场景会非常有用。

第五类是研究、自动化与协作扩展类。社区 collection
的价值主要体现在这里。它给出 13 个分类、173 个技能，尤其在 Writing &
Research、Collaboration & Project Management、Utility &
Automation、Security & Testing、Learning & Knowledge
等方向展示了长尾生态。例如 Meeting Insights Analyzer、Notion Integration
Skills、dispatching-parallel-agents、remembering-conversations
等技能，体现出 Skills 正在从单点工具走向复杂工作流的协调器。

第六类是知识、记忆和工作上下文类。虽然官方仓库没有直接把"记忆系统"作为独立顶层分类，但从社区
collection 可以看到 remembering-conversations、commands、personal
workflow 类技能已经出现。这说明 Skills
正在向"把上下文工作方式固化成能力"的方向发展。对企业员工 Agent
来说，这一层未来会非常关键，因为员工真正需要的并不只是"帮我写文档"，而是"理解我的工作上下文并持续帮助我完成任务"。

从实用性看，企业最值得优先落地的仍是前三类：文档与内容生产、企业规范与沟通、研究/自动化/协作扩展。它们既贴近员工工作，又容易形成可控的
ROI。

# 四、基础调用关系：Skills 并不是平铺罗列，而是存在明显的层间依赖

如果只看仓库目录，很容易把 Skills
理解成一组并列插件；但从工程视角看，它们更像分层服务和组合单元。

最底层可以称为"执行与载体层"。这一层包括 docx、pdf、pptx、xlsx
这类围绕具体文件格式和载体进行处理的 Skills，以及 claude-api
这种围绕接口调用的技能。它们直接作用于文件、表格、演示稿、PDF 或
API，是最接近 I/O
的能力层。没有这一层，上层很多技能只能停留在泛化文本生成。

第二层可以称为"表现与格式层"。典型技能包括
brand-guidelines、theme-factory、canvas-design、frontend-design。它们不一定直接决定任务的业务逻辑，但决定输出物的表现形式、视觉风格、品牌一致性和交互质量。在传统软件里，这一层类似模板引擎、主题系统、前端组件库和样式规范。

第三层是"领域任务层"。例如
internal-comms、doc-coauthoring、webapp-testing、mcp-builder。它们把底层载体能力和格式能力组织起来，面向某类任务提供操作手册式、流程式或专用领域式的技能。可以理解为面向用例的服务封装。

第四层是"复合编排层"。这在官方仓库中更多通过技能组合来体现，在社区
collection 中则更明显。例如 dispatching-parallel-agents、meeting
insights、Notion integration、automation commands 等技能，说明一个高阶
Skill
完全可以调用多个下层能力：先检索信息，再抽取结构化内容，再按品牌模板生成文档，最后投递到协作系统。这个层级本质上就是
agent workflow composition。

第五层是"产品与组织治理层"。这不是某一个 Skill
文件夹，但它是企业落地不可缺失的一层。它决定哪些 Skills
能被谁使用、调用什么模型、访问哪些资源、保存哪些记忆、何时触发审批、如何被审计，以及如何对员工展示为简单产品体验。换句话说，真正面向员工的产品，不能直接把几十个
Skills 暴露给用户，而应把它们封装成可理解、可选择、可治理的功能层。

从调用关系上看，一个典型的企业任务可能遵循如下链路：用户提出需求 →
产品路由器识别场景 → 调用知识/搜索/记忆型 Skill 获取上下文 →
调用文档或数据型基础 Skill 处理载体 → 调用品牌/模板/规范型 Skill
统一输出 → 如有必要再调用协作/系统连接 Skill 投递结果。这说明 Skills
的真正意义在于形成"有层次的编排图"，而不是孤立的工具集合。

# 五、高阶使用模式：从单 Skill 调用到企业级复合工作流

Anthropic Skills
的高阶价值体现在"组合使用"而不是"单个技巧"。从企业员工使用场景出发，可以归纳出四种高阶模式。

第一种模式是"基础技能 + 规范技能"的组合。最简单的例子是先用 docx
生成文档，再用 brand-guidelines 或 theme-factory
对内容进行品牌化和样式统一。这样的模式非常适合企业报告、内部通知、市场材料和培训文档。它体现的不是更强的模型，而是更稳定的输出治理。

第二种模式是"知识获取 + 结构化提取 + 载体输出"的组合。例如先从多份 PDF
中提取信息，再汇总到 xlsx，最后输出为 PPT 或 Word
文档。这个流程中，pdf、xlsx、pptx、docx
不是替代关系，而是典型的链式协作关系。对员工来说，这其实就是"把调研变成汇报材料"的日常工作。

第三种模式是"领域工作流
Skill"。这种模式不是简单串联，而是把多个步骤封装成一个上层技能。例如
doc-coauthoring 本质上就是对文档共创流程的封装；internal-comms
则更像对组织沟通写作方式的封装。企业可以进一步将合同审阅、需求梳理、周报生成、会议纪要整理、政策问答等工作流封装成上层
Skills，使普通员工不必理解底层调用。

第四种模式是"自举与扩展"。官方仓库中的 skill-creator 和 mcp-builder
很有代表性：一个 Skill 可以帮助生成新的 Skill，或者帮助创建新的 MCP
服务。这意味着 Skills
不只是消费层能力，也能成为能力扩展层本身。对于企业平台团队来说，这尤其重要，因为它意味着企业可以在统一范式下不断扩展自己的
Agent 能力边界。

结合社区 collection
观察，高阶使用还呈现出两个趋势。其一是"从单用户任务转向协作任务"，例如并行子
agent
分发、会议洞察、项目管理集成。其二是"从即时调用转向持续上下文"，例如
remembering-conversations 这类记忆相关技能。这两个趋势意味着企业员工
Agent 的未来，不只是一次性完成某个文档，而是持续参与员工的工作流。

# 六、官方仓库与社区 collection 的关系：标准层、样板层、生态层

把 anthropics/skills 与 claude-skills-collection
放在一起看，可以得到比较清晰的生态结构。

官方仓库的角色更接近"标准层 + 参考实现层 +
部分生产能力示例层"。它告诉开发者 Skills
是什么、应该怎么组织、在哪些产品里可用、以及一组官方认可的高质量技能样例长什么样。它的价值在于权威和规范性。

社区 collection 的角色更接近"生态目录层 + 发现层 + 长尾补充层"。其
README 说明该仓库收集并组织了所有公开可用的 Claude
Skills，包含官方和社区技能，并给出 173 个技能、13
个分类。这一集合的优势是广度，帮助我们看见官方仓库之外的真实使用方向。例如营销与
SEO、职业求职、学习与知识、媒体内容、协作项目管理、安全测试等方向，在官方仓库中并不集中展示，但在社区层已经出现较多尝试。

两者并不是替代关系，而应被理解为上下游关系。官方定义 Skill
结构与主流模式，社区将其扩展到更多实际场景。对企业而言，最合理的用法不是盲目从社区
collection
里挑技能上线，而是：先以官方仓库为"能力模型和质量基线"，再把社区
collection 作为"创新雷达和模式库"。任何进入企业环境的社区
Skill，都应经过来源、许可、代码质量、安全性、模型依赖、上下文边界和效果稳定性评估。

从企业设计的角度看，这种关系很像软件生态里的"标准库 +
开源社区包管理"。标准库稳定、可信、适合做基础；社区包丰富、创新快，但需要筛选和治理。因此，如果要为员工设计
Skills
分层架构，底层宜偏官方与自研，中高层可参考社区模式，上层产品层则必须完全企业化。

# 七、面向企业员工日常使用的 Skills 分层架构设计

结合官方仓库与社区 collection 的观察，面向企业员工的 Skills
架构建议分为五层。

第一层是"基础执行层"。这层负责文件、表格、演示稿、PDF、API、浏览器、消息和代码执行等原子能力。官方的
docx、pdf、pptx、xlsx、claude-api
可以作为这一层的典型参考。企业如果构建自己的
Agent，应把这层做成可控、可测试、可观察的基础服务，而不是散落在各个业务
Skill 中。

第二层是"通用办公能力层"。这层围绕员工高频工作，封装摘要、比对、抽取、格式转换、会议纪要、周报、邮件、演示文稿生成、知识问答、表格整理等能力。它应大量依赖基础执行层，但对用户暴露的是"任务语义"，而不是文件格式细节。这一层相当于员工视角的通用工具箱。

第三层是"企业知识与规范层"。这层把品牌规范、内部措辞规范、文档模板、审批逻辑、业务词典、知识库连接、权限策略、组织结构、系统别名等注入到
Agent 中。官方 brand-guidelines、internal-comms、doc-coauthoring
已经是很好的样板。对联想这类企业而言，这一层的作用甚至大于文档技能本身，因为只有纳入企业语境，Agent
才真正有组织价值。

第四层是"复合流程与编排层"。这层由多个 Skills
组合而成，面向完整任务。例如"从多份技术文档和会议纪要中提炼要点 →
生成英文汇报邮件 → 附带数据表和PPT摘要"；又如"读取合同 PDF → 抽取字段 →
写入表格 → 生成审阅意见 →
提交到审批系统"。这层需要工作流引擎、模型路由、错误恢复和阶段性检查，是
Agent 体验差异化的关键。

第五层是"治理与产品层"。这一层决定员工实际如何使用
Agent。它包括技能市场、权限策略、调用白名单、模型策略、记忆策略、审批和审计、成本控制、质量回放、反馈闭环和
UI 封装。没有这一层，Skills 再多也只是开发者工具箱；有了这一层，Skills
才会变成企业产品能力。

需要特别强调的是，记忆和上下文管理应横切所有层级。企业 Agent
的记忆至少应分为：即时会话上下文、任务级工作记忆、用户偏好记忆、团队共享知识和系统状态记忆。不同类型记忆的保留时间、访问权限和写入方式应完全不同。否则
Skills 越多，系统越容易混乱。

# 八、基础调用图：一个企业任务如何经过多层 Skills 完成

下面用一个典型员工任务来说明调用关系。场景为：员工希望根据几份会议纪要、一个
Excel 数据表和两份 PDF
技术资料，生成一份带品牌风格的英文管理汇报，并保存为 Word 和 PPT。

第一步，路由层识别任务类型：这是"多源信息综合 +
管理层材料输出"任务。系统调用检索/记忆能力定位相关文件，同时调用权限层确认用户可访问这些资源。

第二步，基础执行层启动。pdf Skill 从 PDF
中抽取文本、表格和关键元数据；xlsx Skill 读取表格结构与计算结果；必要时
docx / pptx 技能准备载体模板。

第三步，通用办公层进行结构化处理。信息抽取 Skill
归纳技术资料重点，会议纪要 Skill 整理 action items 和结论，表格解析
Skill 识别关键数据趋势，并统一形成结构化中间结果。

第四步，企业知识与规范层介入。brand-guidelines / internal-comms
类技能将内容改写为符合企业高管邮件风格、术语标准和品牌格式的表达。若企业有特定模板或汇报框架，也在此阶段套用。

第五步，复合流程层生成最终产出。docx 负责输出正式 memo，pptx 负责生成 5
页管理摘要，必要时系统再调用邮件或协作连接技能将结果发送或保存。

第六步，治理层记录全过程，包括所用
Skills、所用模型、中间数据摘要、审批节点、输出位置和用户反馈。若用户修订了结果，这些反馈还能沉淀为后续工作流的优化依据。

从这个例子可以看到，Skills
架构最怕两种错误：一种是全部堆在一个巨大的"万能
skill"里；另一种是全部变成原子工具让用户自己拼。前者难维护，后者难使用。正确做法是中间有清晰的层次和组合边界。

# 九、高阶设计建议：企业 Skills 体系应如何治理

如果企业希望把 Skills 作为员工 Agent
的基础服务层，至少需要建立七项治理机制。

第一，来源治理。技能应区分官方、内部自研、可信第三方和实验性社区来源，不同来源对应不同上线门槛。

第二，版本治理。每个 Skill
应有版本号、变更记录、回滚方案和适配矩阵。尤其当技能依赖脚本、外部 API
或模型能力时，更需要版本约束。

第三，权限治理。Skill
的权限不应只依赖用户权限，还要考虑技能本身的敏感等级。例如读取本地文件、连接企业知识库、发邮件、改文档、调用外部搜索，其风险级别完全不同。

第四，模型治理。不同 Skill
可以绑定不同模型策略。基础抽取和格式转换可用低成本模型，高风险总结和复合编排可用更强模型。模型路由是控制成本和质量的关键杠杆。

第五，记忆治理。Skill
是否能读写长期记忆、是否能访问共享记忆、如何引用历史任务，都应明确界定。长期记忆尤其需要审计和用户可见性。

第六，可观测性与审计。对企业环境来说，最少应记录调用链路、失败原因、模型选择、输入输出摘要和外部连接行为。否则问题难以排查，也难以建立信任。

第七，反馈闭环。Skill
不应是静态资产。员工的修订、拒绝、收藏、复用、替代路径，都应成为技能优化与路由改进的数据基础。

这一治理体系的核心思想是：企业 Skills
不是"装更多技能"，而是"让技能以可管理的方式工作"。从长远看，真正构成竞争力的不是技能清单，而是围绕技能形成的平台机制。

# 十、对联想等大型企业的启示

对于联想这类大型企业，Anthropic Skills 体系至少带来四点启示。

第一，Agent
平台建设不能只从模型能力出发，还要从"技能层"出发。模型决定智能上限，Skills
决定可复用性、稳定性和企业落地形态。

第二，办公文档能力非常适合作为第一批基础服务。官方对
docx、pdf、pptx、xlsx 的投入说明，这类能力是员工 Agent
最接近价值兑现的入口。因为它们直接对应日常工作载体。

第三，企业规范必须以 Skill 的方式沉淀，而不是散落在 prompt
里。品牌规范、沟通规范、模板规范、审批规范、知识字典与业务术语都应进入企业技能层。这样才能在不同应用、不同模型、不同场景间保持一致体验。

第四，社区生态值得观察，但不宜直接等同于生产可用能力。社区 collection
的真正价值不是"拿来即用"，而是展示长尾模式和创新方向。企业应从中抽取有用模式，结合自己的治理架构做再实现。

如果进一步面向员工个人 Agent
设计，建议把产品层能力控制在少数高频入口：文档助手、知识助手、汇报助手、会议助手、数据助手。每个入口背后再调用统一的
Skills 层。这样既能隐藏技术复杂性，又能逐步扩展能力边界。

# 结论

Anthropic Skills 体系的重要性，在于它把 Agent
开发从"围绕模型做一次性推理"推进到了"围绕技能做模块化工程"。官方仓库给出了权威定义和高质量样板，社区
collection 展示了生态宽度和未来方向。两者结合起来，可以帮助企业把 Skills
理解为 Agent 的基础服务层，而不只是一些零散插件。

对于企业员工日常使用的
Agent，最合理的路线不是罗列大量技能，而是设计一套分层结构：底层是基础执行技能，中层是通用办公与企业规范技能，上层是复合流程与工作流技能，最上层则是治理与产品层。只有这样，Skills
才能既复用模型能力，又被组织以可控方式吸收。

因此，若将 Skills 视为企业 Agent
平台的"服务编排单元"，那么未来企业的核心能力不再只是拥有更强模型，而是是否能构建出一套高质量、可治理、可演进、可组合的
Skills 体系。Anthropic 的仓库让我们看到了这一范式的起点，社区 collection
则让我们看到了它的外延。对企业来说，下一步不是继续追问"有哪些
Skill"，而是开始回答"我们需要一套什么样的 Skills 架构"。
