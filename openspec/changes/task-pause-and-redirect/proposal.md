# 任务暂停与重定向 — 提案

## 1. 问题

当 AI 在执行多步骤任务（tool loop）时，用户目前只能：

- **什么都不做**：等任务跑完
- **点停止**：完全终止，丢失上下文，需从头重说

缺少中间态：**在任意工具调用边界暂停，人工看结果、给新指令，让 AI 按调整后的方向继续**。

### 典型场景

| 场景 | 用户想做什么 |
|------|-------------|
| AI 在写代码但方向跑偏 | 暂停，告诉它「换用另一个库」，继续 |
| AI 执行了前3步，第4步之前想确认 | 暂停，批准或修改，继续 |
| AI 搜集资料到一半，发现关键词错了 | 暂停，纠正关键词，继续 |
| AI 写文档，风格不对 | 暂停，说「换成更简洁的语气」，继续 |

---

## 2. 核心设计决策

### 决策 A：暂停点在哪？

**选择：工具调用之间**（tool loop 的 iteration 边界）

理由：
- 工具调用内部（bash 执行中）强行中断会留下"半完成"状态（文件只写了一半、命令未退出）
- 工具边界是天然的检查点：每个工具刚完成时，状态是一致的
- 实现上复用现有 `abort` 机制：在工具结束通知（`tool_result` 事件）后暂停，不发起下一轮 LLM 调用

### 决策 B：暂停机制实现

**不引入新的 Core 后端机制**，完全在前端（Frontend）实现：

1. 前端监听 `tool_result` 事件，当用户之前点击了"暂停"按钮时：
2. 调用现有的 `abort()` → Core 中止（等当前工具执行完的话需要在工具完成后 abort，所以需要一个"暂停意图"标志）
3. 实际上：**abort 在当前工具刚完成、下一轮 LLM 尚未发起时触发** = 干净的边界

由于现有 abort 是立即的（不等工具完成），需要区分两种中止：

| 类型 | 触发时机 | 行为 |
|------|---------|------|
| **停止**（现有）| 立即 | 中止 LLM/工具，标记 `⏹ 已中止` |
| **暂停**（新增）| 工具完成后 | 中止下一轮 LLM，进入 `paused` 状态，输入框改为"重定向"模式 |

### 决策 C：重定向消息如何构造？

**选择：把用户指令作为新 user 消息，由 AI 自行结合上下文**

Core 的 session 历史已经包含了：
- 用户原始任务
- 所有已执行工具的 call + result（通过 `messages` 数组）

用户只需输入新指令，如「换用 lodash 库继续」，AI 拿到完整历史后自然能理解已完成的部分和新方向。

不需要在前端额外构造"任务摘要"，避免信息丢失/摘要不准确的问题。

### 决策 D：暂停后 UI 状态

引入前端状态 `paused`：

```
streaming: false → 标准空闲（可发新消息）
streaming: true  → 正在运行
paused: true     → 暂停中（任务已停在检查点，等用户指令）
```

`paused` 时的 UI 区别于普通空闲：
- 输入框 placeholder 变为「输入指令继续任务…（Enter 发送）」
- 输入框顶部出现一条暂停提示横幅
- 发送后自动清除 `paused` 状态
- 也可以点「取消」回到普通空闲（不继续任务）

## 3. 用户流程

```
用户发送任务
    ↓
AI 开始执行：tool_start → tool_result → tool_start → ...
    ↓
用户点击「⏸ 暂停」按钮
    ↓
前端设置 pauseIntent = true
    ↓
当前工具执行完，tool_result 事件到达
    ↓
前端检测 pauseIntent → 调用 abort()
    ↓
状态进入 paused = true
    ↓
UI 显示「已暂停，已完成 N 个工具调用」
    ↓
用户在输入框输入指令（或什么都不说、只按发送继续）
    ↓
发送 → 前端清除 paused，以新消息继续任务
```

---

## 4. 技术约束与边界

### 约束 1：暂停只在 tool_result 之后生效

如果 AI 正在做纯文本回复（无工具调用），暂停按钮行为降级为停止。

### 约束 2：Core 侧无感知

Core 不知道"暂停"的存在，对它来说只是：
1. 收到 `POST /chat/stream` 请求（原始任务）
2. 收到 `POST /chat/abort`（abort 掉）
3. 收到新的 `POST /chat/stream` 请求（重定向消息）

session 历史保存在 Core，两次请求共享同一个 `sessionKey`，上下文完整保留。

### 约束 3：暂停意图丢失的处理

如果 pauseIntent 设置后，AI 的下一个事件不是 `tool_result` 而是 `done`（任务刚好完成了），则：
- 正常完成，清除 pauseIntent
- 不进入 paused 状态
