# Design Draft: 模型自适应工具护栏（Model-Adaptive Tool Guard）

> 状态：第一阶段（Answer Evidence Guard）已实施  
> 背景：强模型（如 `GPT-5.4`）在多数场景能自然完成"理解意图 → 调工具 → 给结论"，但中等模型在相同提示下会出现"应查未查、应证未证"的不稳定行为。

---

## 1. 问题定义

当前系统里，用户会提出两类非常不同的问题：

1. **纯语言问题**：解释、总结、对比、建议、翻译
2. **需要证据的问题**：
   - `推到 git 了么？`
   - `改好了么？`
   - `编译过了吗？`
   - `服务起来了吗？`

对于第 2 类问题，理想行为不是“模型猜一个最像的答案”，而是：

- 结合上下文判断这是**状态核验**
- 主动调用工具获取当前事实
- 再给出带证据的回答

问题在于：

- 强模型通常能自然做到这一点
- 中等模型有时会直接复述上下文、做概率性推断，跳过工具调用
- 单纯继续加强 `system prompt` 只能**提高概率**，不能提供一致性保证

因此需要一个折中设计：

> **不压制强模型的自然规划能力，同时为关键断言提供最小护栏。**

---

## 2. 设计目标

### 2.1 Goals

- 对强模型保持“自然流畅”的体验，不把所有问题都规则化
- 对关键断言场景提供稳定兜底，避免“没查却说查了”
- 护栏应尽量后置、轻量，而不是前置地把所有问题路由成规则引擎
- 护栏强度应可按模型能力分层配置
- 未来可扩展到 Git、编译、文件修改、服务状态、HTTP 验证等场景

### 2.2 Non-Goals

- 不把整个 agent 改造成硬编码工作流引擎
- 不要求每条消息都先跑工具
- 不要求弱模型也达到强模型级别的“自然智能”
- 不在本设计中规定具体 UI 改动

---

## 3. 核心思路

采用三层结构，而不是“全靠 prompt”或“全靠硬规则”。

### 3.1 第一层：Prompt 继续做软约束

继续保留并强化如下信息：

- 当前是 Windows 桌面智能助理
- 可以操作各种工具
- 对“已经完成”“已经修改”“已经推送”这类结论，应优先基于工具证据
- 没有真实 `tool_result` 时，不应伪造执行结论

这层适合强模型，成本最低，也最自然。

### 3.2 第二层：关键断言的后置证据校验

不提前强迫所有查询都走工具，而是在**最终回答生成后**做一次轻量校验：

如果模型回答里包含“已经推送 / 尚未推送 / 已修改 / 已执行 / 服务已启动 / 编译已通过”等**事实性断言**，但本轮没有对应的工具证据，则：

- 不直接放行该回答
- 进入“证据缺失”分支
- 给模型一个最小纠偏：
  - 要么自动补一次工具检查
  - 要么将回答改写成“我还没有实际核验，需要先检查”

这层是本设计的关键：

> **尽量不干预模型如何思考，但对“没有证据却给事实结论”的结果进行兜底。**

### 3.3 第三层：高风险动作必须显式授权

对带副作用的动作，仍保持保守：

- `推到 git 了么？` → 先检查，不自动 `push`
- `帮我 push 一下` → 才允许执行 `push`
- `删掉这个文件么？` → 先确认
- `帮我重启服务` → 明确执行型意图后再操作

也就是说：

- **状态核验** 可以自动查
- **副作用动作** 不应由“状态问句”隐式触发

---

## 4. 为什么不建议“全硬规则”

如果把“有没有 / 是不是 / 还在吗 / 推了么”全部前置成硬编码规则，会有几个明显问题：

1. **压制强模型能力**  
   强模型本来就能自然理解上下文、主动查证，过重规则会让它显得机械。

2. **交互变笨重**  
   所有状态问题都先进入固定工作流，会增加不必要的工具调用、延迟和 token 开销。

3. **维护成本高**  
   规则表会越来越大，而且很难覆盖所有自然语言变体。

4. **误伤正常对话**  
   比如“你觉得我是不是更适合 TypeScript？”这种“是不是”并不是状态核验。

因此本设计明确选择：

> **以强模型为默认主路径，用最小后置护栏保护产品可信度。**

---

## 5. 模型分层策略

不同模型能力不同，护栏强度也应不同。

### 5.1 Tier A：高能力模型（如 `GPT-5.4`）

策略：**Prompt 主导 + 轻量后置校验**

- 默认相信模型能做好任务规划
- 不做前置强制路由
- 只在出现“无证据断言”时兜底

适用原因：
- 这类模型通常能理解“推到 git 了么？”是一个需要核验的问题
- 其主要风险不是“不懂”，而是偶尔漏调工具

### 5.2 Tier B：中等模型（如某些 4o / M2.7 类模型）

策略：**Prompt + 后置校验 + 少量前置提示增强**

- 仍不做重规则化
- 但会对“状态核验类问题”增加更强的提示修正
- 如果首次回答未查证，则自动引导补查

### 5.3 Tier C：低服从性模型

策略：**可选前置轻路由**

- 对少数高频场景（Git 状态、编译状态、服务状态）增加意图识别
- 命中后优先生成工具计划
- 仍保留后置校验

---

## 6. 场景分类

### 6.1 必须有证据的回答类型

这类回答在产品上不应允许“纯猜”：

- **Git 状态**：是否已 commit / push / merge
- **文件改动状态**：是否已经写入 / 修改 / 删除
- **命令执行结果**：是否执行成功 / 失败
- **编译测试结果**：是否通过
- **服务状态**：是否启动、端口是否可访问
- **HTTP/API 验证**：某端点是否返回预期结果

### 6.2 可纯语言回答的类型

- 为什么这样设计
- 有哪些方案
- 哪个方案更合适
- 帮我总结一下
- 这段代码大概在做什么

### 6.3 状态核验 vs 执行动作

需要明确分开：

| 用户说法 | 类型 | 推荐行为 |
|---|---|---|
| `推到 git 了么？` | 状态核验 | 自动检查，不自动 push |
| `帮我 push 到 git` | 执行动作 | 直接执行（如用户已明确） |
| `改好了么？` | 状态核验 | 自动检查改动是否存在 |
| `帮我把这个改好` | 执行动作 | 执行修改 |

## 7. 推荐架构

### 7.1 方案 A：Answer Evidence Guard（推荐起步方案）

在 `runner` 的最终回答输出前，加入一个轻量守卫：

#### 输入
- 用户原始消息
- 本轮执行过的工具集合
- 工具结果摘要
- 模型最终回答
- 当前模型档位（A/B/C）

#### 输出
- `pass`：允许直接输出
- `rewrite`：改写为“尚未实际核验”
- `retry-with-tools`：要求模型补一次工具核验

#### 优点
- 最少侵入
- 对强模型最友好
- 能覆盖“偶尔漏查”的关键问题

#### 缺点
- 它是结果校验，不是前置规划
- 对极弱模型只能兜底，不能提升太多主动性

### 7.2 方案 B：Intent-Aware Query Classifier（可选增强）

只对少数高价值场景做分类：

- `status_check`
- `action_request`
- `pure_conversation`

用于在中等模型下增强稳定性。

注意：
- 这个分类器不应变成大而全的规则系统
- 只服务于非常明确的产品风险点

### 7.3 方案 C：Structured Tool Policies（长期方案）

为工具和问题类型增加结构化元数据，例如：

```ts
questionPolicy = {
  git_status: { requiresEvidence: true, sideEffect: false },
  git_push: { requiresEvidence: false, sideEffect: true, needsConfirmation: true },
  test_result: { requiresEvidence: true, sideEffect: false },
}
```

这是长期可演化方向，但不建议一开始就做得太重。

## 8. 以“推到 GIT 了么？”为例

### 理想流程

1. 模型结合上下文，理解用户是在问“当前仓库是否已 push”
2. 主动调用 Git 相关工具或 `bash` 检查：
   - 工作区是否干净
   - 本地是否领先远端
   - 最近提交是否已同步
3. 输出结论：
   - 已推送 / 未推送
   - 最新 commit 是什么
   - 若未推送，可补一句：`要不要我现在帮你推？`

### 不理想流程

- 直接复述上文“我已经推了”
- 没有任何工具结果却说“已经推送成功”
- 用户只是问状态，却自动执行 `push`

### 该场景的产品判断

`推到 GIT 了么？` 是：
- 一个 **状态核验问题**
- 允许自动查证
- 不允许无证据断言
- 不应隐式升级成自动 `push`

## 9. 与现有设计的关系

当前仓库已存在一份相关设计：

- `openspec/changes/tool-execution-proof-guard/design.md`

那份设计更关注：
- “没有工具结果却宣称已经执行”的**硬性拦截**

而本设计更关注：
- **如何在强模型优先的前提下，避免系统变得过度规则化**
- **如何按模型档位自适应调整护栏强度**

两者关系是：

- `tool-execution-proof-guard`：偏“执行证据”
- `model-adaptive-tool-guard`：偏“模型能力分层 + 护栏策略”

推荐把本设计视为前者的上层策略补充，而不是替代。

## 10. 开放问题

这些问题建议在正式立项前讨论清楚：

1. **证据缺失时优先改写，还是优先自动补查？**  
   - 改写更稳，但不够智能
   - 自动补查更智能，但会增加延迟

2. **哪些工具结果算“足够证据”？**  
   例如 Git 状态是否必须看 `upstream`，还是 `git status` 就够

3. **护栏强度是否要跟模型绑定？**  
   比如 `GPT-5.4` 用轻护栏，`MiniMax-M2.7` 用中护栏

4. **是否引入专门的“状态核验问题分类器”？**  
   这会提高稳定性，但会增加系统复杂度

5. **是否应优先使用结构化 Git 工具，而不是通用 `bash`？**  
   这通常会更稳，也更容易做证据校验

---

## 11. 当前建议

如果只做一版最小可用设计，我建议：

### 第一阶段
- 保持强模型主路径不变
- 不上重型前置规则
- 增加“无证据断言”的后置校验
- 高风险副作用仍要求显式意图

### 第二阶段
- 按模型档位调护栏强度
- 对 Git / 编译 / 服务状态补少量查询分类

### 第三阶段
- 收口高频场景到结构化工具策略

一句话总结：

> **不是“要不要写硬规则”，而是“在不打断强模型自然能力的前提下，给产品可信度加一层最小护栏”。**
