# 任务感知 — 提案

## 1. 问题

Equality 现在是"工具调用型 agent"——用户给指令，它执行。这本身没问题，但有三类高频失败场景：

### 场景 A：任务模糊，直接开跑
```
用户：帮我整理一下项目
→ Agent 直接开始读文件、执行操作
→ 用户意思可能是"整理代码结构"、"整理文档"、"整理 Git 历史"……
→ 执行完发现方向全错，浪费时间和 token
```

### 场景 B：长任务中途失控，用户不知道进展
```
用户：帮我分析这个月的销售数据并生成汇报
→ Agent 静默执行 8 轮工具调用
→ 某一步出错，前面的工作白费
→ 用户不知道它在做什么，无法提前纠偏
```

### 场景 C：任务完成后没有总结
```
Agent 执行完 12 步工具调用
→ 最后输出一段流水账（"第1步我做了…第2步…"）
→ 用户需要自己读完才知道"到底搞定了没有"
```

---

## 2. 目标

提升"任务感知能力"，让 Agent 在三个关键节点表现得像一个有经验的执行者：

1. **任务前**：主动澄清歧义，而不是猜测后直接执行
2. **任务中**：长任务开始前输出执行计划，让用户知道接下来要做什么
3. **任务后**：以结构化摘要收尾，而不是工具调用流水账

---

## 3. 方案概述

### Step 1：主动澄清（Pre-task Clarification）

当用户请求模糊、关键参数缺失时，Agent 在执行任何工具调用之前，先提出 1-2 个最关键的问题。

**触发条件**：
- 请求中缺少必要的目标/范围/路径信息
- 同一请求有多种截然不同的执行方向

**约束**：
- 最多问 2 个问题，不超过
- 如果可以合理推断，不要问
- 不要问可以通过工具探查的问题（比如"你的项目目录在哪里"，直接用 list_directory 看）

### Step 2：执行前计划（Pre-execution Plan）

对于需要 3 步以上工具调用的任务，在第一个工具调用前，先用文字输出执行计划。

格式：
```
📋 执行计划：
1. 读取 xxx 文件，确认数据结构
2. 用 Python 脚本处理数据
3. 生成 HTML 报告写入目标路径
```

**触发条件**：Agent 判断任务需要 ≥ 3 个工具调用步骤

**约束**：
- 计划尽量简短（每步一行，不解释细节）
- 计划是预测，允许执行中调整，不需要严格对应

### Step 3：执行后摘要（Post-execution Summary）

任务完成后（最后一个工具结果返回，进入最终 LLM 输出时），以结构化方式输出：

格式：
```
✅ 完成

**做了什么**：生成了月度销售分析报告

**结果**：
- 报告路径：C:/reports/2024-03.html
- 数据范围：2024年3月1日～31日，共 1,203 条记录
- 关键发现：华东区环比增长 23%

**注意事项**：原始数据中有 12 条缺失 SKU，已跳过
```

**触发条件**：本次交互调用了 ≥ 2 个工具（纯对话不触发）

---

## 4. 实现方式

三个功能全部通过 **system prompt 追加指令** 实现，不改动 runner 逻辑。

- 改动文件：`packages/core/src/agent/system-prompt.ts`
- 改动量：在 `buildSystemPrompt()` 输出中追加一个"任务感知规则"块
- 不涉及：runner.ts、前端代码、API 接口

---

## 5. 不做什么

- 不做结构化的"规划阶段"（单独一次 LLM 调用做规划）——成本高，对简单任务是负担
- 不做用户确认步骤（"确认后再执行"的流程）——增加交互摩擦
- 不做任务进度的实时 UI 展示——属于前端改动，不在此范围
- 不做强制等待澄清的阻塞逻辑——Agent 自行判断是否需要问
