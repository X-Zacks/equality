# 任务感知 — 设计文档

## 1. 架构决策

### 为什么用 system prompt 而不是改 runner 逻辑？

三个备选方案：

| 方案 | 描述 | 优点 | 缺点 |
|---|---|---|---|
| **A. System prompt 追加规则** | 在 `buildSystemPrompt()` 加指令，LLM 自行判断何时澄清/计划/摘要 | 零 runner 改动；LLM 有上下文判断力，比硬编码规则灵活 | 依赖 LLM 遵从指令 |
| B. Runner 硬编码触发 | 在 runner 里检测 toolCallCount，达到阈值后注入特殊 message | 确定性强 | 改动大；注入消息会影响 token 计量和上下文窗口 |
| C. 独立规划阶段 | 任务开始前额外调用一次 LLM 做规划 | 规划质量高 | 每次任务多一次 LLM 调用，成本高；简单任务是负担 |

**选择方案 A**，原因：
1. 改动量最小（只改 1 个函数，约 20-30 行）
2. LLM 的上下文理解能力完全可以胜任这种判断
3. 灵活性高：LLM 能根据具体情境决定是否触发，不需要维护硬编码阈值

函数 `buildSystemPrompt()` 的输出末尾追加**任务感知规则块**。

## 3. 任务感知规则块内容设计

### 3.1 主动澄清规则

核心约束：
- 触发条件：关键信息缺失 **且** 无法通过工具探查
- 最多 2 个问题
- 可以用工具探查的，不问，直接查
- 有合理默认时，声明假设后直接执行

指令文本（加入 system prompt）：
```
## 任务感知规则

### 1. 执行前澄清
当用户请求存在关键歧义（同一请求有多种截然不同的执行方向）或缺少无法推断的必要信息时：
- 在调用任何工具之前，提出 1-2 个最关键的问题
- 只问无法通过工具自行查明的信息（目录结构、文件内容等可以直接用工具看）
- 如果可以合理推断或有明显默认解释，不要打断用户——在开头说明你的理解假设，直接执行
```

### 3.2 执行前计划规则

核心约束：
- 触发条件：预计需要 ≥ 3 步工具调用
- 计划在第一个工具调用前输出（作为 delta 文本）
- 格式固定（📋 + 编号列表）
- 输出后立即执行，不等用户回复

```
### 2. 执行前计划
当任务需要 3 个或更多工具调用步骤时，在调用第一个工具前用文字输出执行计划：

📋 执行计划：
1. [第一步简述]
2. [第二步简述]
3. [第三步简述]
...

每步一行，简洁描述目标（不写工具名称和参数）。输出计划后立即开始执行，不要等待用户确认。
```

### 3.3 执行后摘要规则

核心约束：
- 触发条件：本次调用了 ≥ 2 个工具
- 替代流水账式描述
- 必须包含：状态标识 + 核心结果
- 可选：注意事项

```
### 3. 执行后摘要
当本次完成了 2 个或更多工具调用后，最终回复使用结构化摘要，不要逐步描述每个工具的执行过程：

✅ 完成（或 ⚠️ 完成但有问题）

**做了什么**：[一句话描述核心任务]

**结果**：
- [关键结果条目，如文件路径、数量、关键数据等]

**注意事项**（如有）：[遇到的问题、跳过的内容、用户需要知道的事]

纯对话回答（未调用任何工具）或只调用了 1 个工具时，正常回答即可，不强制使用此格式。
```

## 5. 与现有规则的关系

当前 system prompt 已有的块：
1. 基本身份 + 当前时间/平台/工作目录
2. Windows PowerShell 脚本规则
3. Skills 索引（可选）
4. Skill 沉淀指令

新增的"任务感知规则"块追加在第 2 块之后，Skills 块之前：

```
[身份 + 时间/平台]
[Windows PowerShell 规则]
[任务感知规则]  ← 新增
[Skills 索引]（可选）
[Skill 沉淀指令]
```

## 6. 验证思路

由于是 prompt 改动，验证方式是黑盒测试：

| 场景 | 输入 | 期望行为 |
|---|---|---|
| 澄清触发 | "帮我整理一下项目" | 先问具体方向，不调用工具 |
| 澄清不触发 | "列出当前目录的 Python 文件" | 直接执行，不问 |
| 计划触发 | "分析数据生成 HTML 报告" | 输出 📋 后开始执行 |
| 计划不触发 | "查一下当前目录" | 直接执行，无计划输出 |
| 摘要触发 | 任何 2+ 工具调用任务 | 以 ✅/⚠️ 格式收尾 |
| 摘要不触发 | 纯问答 | 正常文字回复 |
